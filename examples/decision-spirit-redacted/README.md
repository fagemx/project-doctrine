# Example — Decision Spirit (Redacted)

**Status: reference pointer.** The project-doctrine methodology was originally extracted from **Decision Spirit**, a chat-native co-presence decision companion. That project's doctrine is the concrete source from which the six-layer model was generalized.

## Why there's no full redacted version here

Decision Spirit's doctrine is deeply product-specific — lots of bilingual terminology (創造物 / 小生物 / 手帳), specific architecture (Creature / Room / Mecha / Three Books), and scars tied to unreleased product surfaces.

Rather than produce a watered-down "redacted" version that would lose the texture, this example serves as:

1. **A pointer** to the real-world project where the methodology earned its shape
2. **A note on what to borrow** vs **what's decision-spirit-specific**

## What transfers from Decision Spirit's doctrine

- The **6-layer structure itself** (ideology / knowledge / method / SOPs / thinking-modes / heart-methods)
- The **"enter the posture, don't summarize"** instruction
- The **state-snapshot + bootstrap-prompt + provenance** supporting docs
- The idea that **L6 requires scars** — no scar, not L6
- The **apprenticeship check** as a 2-way diagnostic (tests agent + tests doctrine)
- **Framing > model capacity** as a universal AI-project heart-method
- The **reading ladder** (state → ideology → thinking → heart, then on-demand)

## What does NOT transfer from Decision Spirit

- Product-specific L1 entries (co-presence creature, room artifact model)
- Bilingual-Chinese terminology (unless your team is bilingual)
- Specific L6 entries tied to decision-spirit failures
- Architecture-specific layer-2 references

## Where the real doctrine lives

The live doctrine is in the Decision Spirit repo at:
```
docs/skills/decision-spirit-session-experience/
```

If you have access, read it as a worked example. Notice:
- The tight state-snapshot (one screen, dated)
- L1 entries with violation tests
- L6 entries always with a scar and a trigger
- The bilingual couplets that compress lessons

## How this methodology was extracted

1. **Observation:** Decision Spirit accumulated ~3 months of doctrine that was working well
2. **Analysis:** A session-experience skill existed with 6-layer structure that was clearly a reusable pattern
3. **Extraction:** This repo generalizes the form, strips the product-specific content, and produces templates + methodology docs

If you're extracting a similar methodology from your own long-running project, the process is:

1. Notice what you keep re-telling fresh agents
2. Notice what layers of that retelling are stable vs. volatile
3. Abstract the form, keep the content proprietary
