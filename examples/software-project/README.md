# Example — Software Project

A worked example of Project Doctrine for a **generic B2B SaaS app with AI-assisted features**. Voice is deliberately neutral so readers can see the shape without product noise.

## The scenario

- Team of 4 engineers + 2 PMs
- Product has been live ~14 months
- Recently added LLM features (summarization, Q&A over user data)
- Multiple agents (Claude Code, Cursor) actively contributing to the codebase
- Accumulated real scars: one rollback-heavy month, two contentious architecture decisions, a persistent "AI says yes when it should say no" problem

## How to read this example

Start with `SKILL.md`. Then `state-snapshot.md` to see the volatile layer. Then L1 and L6 to see the durable stance.

**The entries are realistic but fictional.** They show the *shape* of a good entry; they do not describe a real product.

## What to borrow

- **Structure** — the SKILL.md and the 12 references layout is directly copyable
- **Entry shapes** — each layer file demonstrates a good entry format
- **Density** — notice how each entry is tight; your doctrine entries should be the same

## What NOT to borrow

- **Content** — the specific L1/L6/FM entries are for this fake product. Your project's entries must come from your project's experience.
- **Language** — the English-only voice here is for clarity. If your team works bilingually, embrace that.
