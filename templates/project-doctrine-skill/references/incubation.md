# Incubation (Candidates)

> Things that might be doctrine material, but we're not sure yet. Parked here until a review pass promotes, archives, or deletes them.

**Rule:** if you don't know whether it's doctrine, don't promote yet. Add here. Decide later.

---

## How to use this file

1. When something surfaces that *might* be doctrine (a suspicion, a half-pattern, an interesting observation), add a candidate entry below.
2. Fill in the four required fields.
3. At each maintenance sweep (weekly / biweekly / event-driven), review candidates:
   - **Still important + pattern confirmed** → promote to failure-memory / taste-examples / L1 / L5 / L6 and delete from this file
   - **Hasn't recurred, no longer interesting** → delete
   - **Historical only** → archive (move to archive notes and mark "historical")

**Candidate entries that sit unmoved for 30 days are presumed noise** and should be deleted unless someone makes a case.

---

## Format

```markdown
## Candidate: <short name>

- **Source:** <where this came from — session, PR, retro, user correction>
- **Why it may matter:** <one line — if you can't write one, it's not candidate material>
- **Not promoted because:** <what's uncertain — recurrence? cost? fit? layer?>
- **Revisit after:** <date OR trigger — e.g. "after next major feature" or "2026-05-20">
```

## Anti-patterns

- **Using this as a long-term parking lot.** Candidates should move out within 30–60 days. If 10+ candidates have been sitting for 3 months, you're avoiding decisions.
- **Candidates with vague "why it may matter".** Force the one-line justification. If you can't write it, it's not a candidate.
- **Never promoting.** If nothing ever graduates from incubation, either the intake is too loose (everything gets parked) or the review cadence is broken.
- **Re-litigating deleted candidates.** Once deleted with reason, leave deleted. The reason protects against re-adding.
